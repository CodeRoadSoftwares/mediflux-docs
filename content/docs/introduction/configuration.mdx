MediFlux is highly configurable, allowing you to tailor the platform to your pharmacy's specific operational requirements. Configuration can be managed through environment variables, a configuration file, or the admin dashboard.

This guide covers the most important configuration options and explains how each setting affects the platform's behavior.

## Configuration File

MediFlux looks for a configuration file at `~/.mediflux/config.yaml` by default. You can specify a custom path using the `MEDIFLUX_CONFIG_PATH` environment variable.

```yaml title="config.yaml"
server:
  port: 3000
  host: 0.0.0.0
  cors:
    origins:
      - https://pharmacy.example.com
    credentials: true

database:
  url: postgresql://mediflux:password@localhost:5432/mediflux
  pool:
    min: 5
    max: 20
  ssl: true

redis:
  url: redis://localhost:6379
  prefix: mediflux

pharmacy:
  default_currency: USD
  tax_rate: 0.08
  require_prescription_verification: true
  controlled_substance_tracking: true
  expiry_alert_days: 90
  low_stock_threshold: 10

accounting:
  fiscal_year_start: "01-01"
  default_payment_terms: 30
  auto_reconciliation: true
  multi_currency: false

auth:
  session_timeout: 3600
  max_login_attempts: 5
  lockout_duration: 900
  password_policy:
    min_length: 12
    require_uppercase: true
    require_number: true
    require_special: true
```

## Environment Variables

Environment variables take precedence over the configuration file. This is useful for managing sensitive values and deployment-specific settings.

| Variable                       | Description                              | Default              |
| ------------------------------ | ---------------------------------------- | -------------------- |
| `DATABASE_URL`                 | PostgreSQL connection string             | —                    |
| `REDIS_URL`                    | Redis connection string                  | `redis://localhost:6379` |
| `MEDIFLUX_PORT`                | HTTP server port                         | `3000`               |
| `MEDIFLUX_SECRET_KEY`          | Application secret for signing tokens    | —                    |
| `MEDIFLUX_ENV`                 | Environment (`development`, `production`)| `development`        |
| `MEDIFLUX_LOG_LEVEL`           | Logging verbosity                        | `info`               |
| `MEDIFLUX_STORAGE_TYPE`        | File storage backend (`local`, `s3`)     | `local`              |
| `MEDIFLUX_STORAGE_PATH`        | Local storage directory path             | `/var/mediflux/uploads` |
| `MEDIFLUX_S3_BUCKET`           | S3 bucket name for file storage          | —                    |
| `MEDIFLUX_S3_REGION`           | AWS region for S3 storage                | `us-east-1`          |
| `MEDIFLUX_SMTP_HOST`           | SMTP server for email notifications      | —                    |
| `MEDIFLUX_SMTP_PORT`           | SMTP server port                         | `587`                |
| `MEDIFLUX_SMTP_USER`           | SMTP authentication username             | —                    |
| `MEDIFLUX_SMTP_PASS`           | SMTP authentication password             | —                    |
| `MEDIFLUX_WEBHOOK_SECRET`      | Secret for signing webhook payloads      | —                    |

<Callout type="warning">
  Never commit sensitive environment variables like `MEDIFLUX_SECRET_KEY` or database credentials to version control. Use a secrets manager or `.env` file excluded from your repository.
</Callout>

## Pharmacy Settings

These settings control pharmacy-specific behavior across the platform.

### Prescription Verification

When `require_prescription_verification` is enabled, prescription medications cannot be dispensed without a pharmacist's digital signature. This is required by regulation in most jurisdictions.

```yaml title="Prescription verification settings"
pharmacy:
  require_prescription_verification: true
  verification_methods:
    - digital_signature
    - biometric
  prescription_validity_days: 365
  refill_limit: 5
```

### Controlled Substance Tracking

MediFlux includes built-in DEA Schedule II-V tracking that maintains a complete audit trail of all controlled substance transactions.

```yaml title="Controlled substance configuration"
pharmacy:
  controlled_substance_tracking: true
  dea_reporting:
    enabled: true
    format: ARCOS
    submission_frequency: monthly
  csos_enabled: true
```

<Callout type="info">
  Controlled substance tracking cannot be disabled if your pharmacy dispenses Schedule II through V medications. This setting is enforced at the application level regardless of configuration.
</Callout>

### Inventory Alerts

Configure how MediFlux monitors and alerts you about inventory conditions:

```yaml title="Inventory alert configuration"
pharmacy:
  expiry_alert_days: 90
  low_stock_threshold: 10
  overstock_threshold: 500
  alert_channels:
    - dashboard
    - email
    - webhook
  reorder_automation:
    enabled: true
    preferred_suppliers: true
    approval_required: true
```

## Advanced Settings

<Accordions>
  <Accordion title="Database Connection Pooling">
    MediFlux uses connection pooling to efficiently manage database connections. The default pool settings work well for most deployments, but high-traffic pharmacies may need adjustments.

    ```yaml title="Pool configuration"
    database:
      pool:
        min: 5
        max: 20
        idle_timeout: 30000
        acquire_timeout: 60000
        reap_interval: 1000
    ```

    For deployments with more than 100 concurrent users, consider increasing `max` to 50 and using PgBouncer as an external connection pooler.
  </Accordion>

  <Accordion title="Rate Limiting">
    API rate limiting protects your MediFlux instance from abuse and ensures fair resource allocation across integrations.

    ```yaml title="Rate limiting configuration"
    api:
      rate_limit:
        enabled: true
        window: 60
        max_requests: 100
        by: api_key
      rate_limit_overrides:
        - api_key: key_internal_pos
          max_requests: 1000
    ```
  </Accordion>

  <Accordion title="Audit Logging">
    MediFlux maintains comprehensive audit logs for compliance and security. Configure what gets logged and where logs are stored.

    ```yaml title="Audit log configuration"
    audit:
      enabled: true
      retention_days: 2555
      log_actions:
        - user.login
        - user.logout
        - sale.create
        - sale.void
        - inventory.adjust
        - prescription.dispense
        - controlled_substance.transaction
      export:
        format: json
        destination: /var/mediflux/audit
    ```

    <Callout type="warning">
      Audit log retention for controlled substance transactions must meet DEA requirements (minimum 2 years). MediFlux enforces a minimum retention of 730 days for these records regardless of configuration.
    </Callout>
  </Accordion>

  <Accordion title="Backup Configuration">
    Configure automated backups to protect your data. MediFlux supports local, S3, and custom backup destinations.

    ```yaml title="Backup configuration"
    backup:
      enabled: true
      schedule: "0 2 * * *"
      retention:
        daily: 7
        weekly: 4
        monthly: 12
      destination:
        type: s3
        bucket: mediflux-backups
        region: us-east-1
      encryption:
        enabled: true
        algorithm: AES-256-GCM
    ```
  </Accordion>
</Accordions>

## Validating Configuration

After making configuration changes, validate your settings before restarting:

```bash title="Validate configuration"
mediflux config validate
```

```json title="Validation output"
{
  "valid": true,
  "warnings": [
    "SMTP not configured - email notifications will be disabled"
  ],
  "errors": []
}
```

Apply changes by restarting the MediFlux service:

```bash title="Restart MediFlux"
mediflux restart --graceful
```

<Callout type="success">
  The `--graceful` flag ensures all in-progress transactions complete before the restart, preventing data loss during configuration changes.
</Callout>
